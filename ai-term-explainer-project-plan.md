# AI 划词与框选解释助手项目计划书

## 1. 项目概述

本项目拟开发一款轻量级桌面工具，帮助用户在学习股票、英语和阅读资料时，快速理解不熟悉的名词、短语、句子或截图中的文字内容。

用户可以通过两种方式触发查询：

- 划词解释：选中可复制文本后按快捷键，自动获取文本并调用大模型解释。
- 框选解释：按快捷键进入截图框选模式，对屏幕区域进行 OCR 识别，再调用大模型解释或翻译。

项目核心定位不是传统翻译软件，而是“随处可用的知识解释助手”。翻译是其中一种能力，重点是根据语境给出简洁、通俗、可操作的解释。

## 2. 项目目标

### 2.1 核心目标

- 支持用户在网页、PDF、股票软件、文档、图片等场景中快速获取解释。
- 支持股票/金融术语、英语单词/短语/句子、普通知识名词的自动识别与解释。
- 使用大模型 API 生成高质量解释，支持 OpenAI-compatible API。
- 保持客户端轻量化，不内置本地大模型，不打包重型 OCR 模型。
- 尽量做到 1 秒内有可见反馈，命中缓存时毫秒级返回。

### 2.2 非目标

第一版不追求成为完整翻译平台，不做复杂全文排版翻译，不做浏览器插件，不内置本地大模型，不提供大型词典系统。

## 3. 目标用户与场景

### 3.1 目标用户

- 股票初学者：阅读财报、研报、行情软件时遇到术语。
- 英语学习者：阅读英文网页、PDF、论文、字幕时遇到单词或句子。
- 泛学习用户：在任意软件中遇到陌生概念，希望快速理解。

### 3.2 典型场景

- 在股票软件中框选“扣非净利润”“市盈率 TTM”，快速得到通俗解释。
- 在英文网页中选中一个短语，按快捷键获得中文释义、语境含义和例句。
- 在扫描 PDF 或图片中框选一段英文，通过 OCR 提取文字并解释。
- 在研报中选中一整句话，获得摘要、翻译和关键概念说明。

## 4. 产品功能范围

### 4.1 MVP 必须功能

1. 全局快捷键
   - 划词解释快捷键，例如 `Ctrl + Shift + E`。
   - 截图框选快捷键，例如 `Ctrl + Shift + S`。

2. 划词取词
   - 用户选中文本后触发快捷键。
   - 程序模拟复制操作并读取剪贴板。
   - 尽量恢复用户原剪贴板内容，减少打扰。

3. 截图框选 OCR
   - 快捷键进入屏幕框选模式。
   - 按下快捷键后，直接以当前鼠标光标位置作为框选起点。
   - 用户继续拖拽鼠标选择屏幕区域，松开鼠标后完成截图。
   - 支持按 `Esc` 取消框选，取消时立即清理遮罩窗口和临时截图资源。
   - 对选区截图进行 OCR。
   - 第一版优先使用 Windows 系统 OCR。

4. 大模型解释
   - 支持 OpenAI-compatible API。
   - 支持配置 API Base URL、API Key、模型名称。
   - 内置默认 prompt，自动判断内容类型。
   - 预置股票/金融、英语学习、通用名词、翻译四类专业化 prompt 模板。
   - 允许用户在设置中修改默认 prompt，但保留一键恢复默认模板。
   - 流式输出结果。

5. 悬浮解释窗口
   - 快捷弹出。
   - 显示识别文本、解释结果、加载状态和错误提示。
   - 支持复制结果。
   - 支持重新生成。
   - UI 结构提前组件化，方便后期美化窗口、切换主题和扩展结果卡片。

6. 本地缓存
   - 相同文本和模式命中缓存时直接返回。
   - 缓存查询结果、时间、类型、模型。
   - 支持基础历史记录。

7. 基础设置
   - 设置 API Key。
   - 设置模型。
   - 设置快捷键。
   - 设置默认解释风格。

### 4.2 后续增强功能

- 生词本/术语本。
- 股票术语专用卡片。
- 英语例句与复习卡片。
- 多模型切换。
- OCR 语言选择。
- 浏览器插件版本。
- PaddleOCR/Tesseract 插件化支持。
- 截图识别后允许用户编辑 OCR 文本再提交。
- 解释模板自定义。

## 5. 核心交互流程

### 5.1 划词解释流程

```text
用户选中文字
  -> 按快捷键
  -> 应用读取选中文本
  -> 弹出悬浮窗
  -> 查本地缓存
  -> 命中则立即显示
  -> 未命中则调用大模型 API
  -> 流式显示解释结果
  -> 写入缓存和历史
```

### 5.2 框选解释流程

```text
用户按截图快捷键
  -> 记录当前鼠标位置作为框选起点
  -> 屏幕进入框选状态并显示半透明遮罩
  -> 用户继续拖拽选择区域
  -> 用户松开鼠标确认选区，或按 Esc 取消
  -> 应用截取选区图像
  -> 调用系统 OCR 提取文字
  -> 弹出悬浮窗并显示识别文本
  -> 查本地缓存
  -> 未命中则调用大模型 API
  -> 流式显示解释结果
  -> 释放遮罩窗口、截图缓冲区和 OCR 临时对象
```

### 5.3 框选交互细节

- 快捷键触发时不要求用户再点击一次屏幕，直接把当前光标坐标作为矩形起点。
- 拖拽过程中实时绘制选区边框和尺寸提示。
- 若选区过小，例如宽或高小于 8px，视为误触并取消。
- 框选完成后立即隐藏遮罩，避免遮罩被截入图片。
- OCR 失败或结果为空时，悬浮窗显示“未识别到文字”，并提供重新框选入口。
- 框选流程必须是一次性任务，结束后释放全部事件监听、图像对象和窗口引用。

## 6. 技术方案

### 6.1 推荐技术栈

- 桌面框架：Tauri
- 前端：React + TypeScript
- 后端能力：Rust/Tauri commands
- 本地存储：SQLite
- 模型接口：OpenAI-compatible Chat Completions API
- OCR：Windows.Media.Ocr
- 截图与窗口：Tauri 插件 + Windows API

### 6.2 轻量化原则

- 不内置大模型。
- 不打包 PaddleOCR 等大型依赖。
- 不启动常驻大型服务。
- 主进程只负责快捷键、剪贴板、截图、OCR、API 请求和缓存。
- UI 使用单个主设置窗口 + 一个悬浮结果窗口。

### 6.3 模块划分

```text
app-shell
  - 应用启动
  - 托盘菜单
  - 全局快捷键
  - 窗口管理

capture
  - 屏幕截图
  - 区域框选
  - 图片裁剪

ocr
  - Windows 系统 OCR 调用
  - OCR 文本清理

selection
  - 模拟复制
  - 剪贴板读取
  - 剪贴板恢复

llm
  - API 配置
  - prompt 构造
  - 流式请求
  - 错误处理

cache
  - 查询缓存
  - 写入缓存
  - 历史记录

ui
  - 悬浮解释窗口
  - 设置页
  - 历史记录页
```

### 6.4 大模型接口抽象

需要提前为不同模型和服务商留出接口。业务层不直接调用某一家模型 API，而是调用统一的 `LlmProvider` 抽象。

```text
LlmProvider
  - id
  - name
  - api_base_url
  - auth_type
  - model
  - supports_stream
  - default_temperature
  - request_timeout_ms
  - send_chat(messages, options)
  - stream_chat(messages, options)
```

第一版优先实现 OpenAI-compatible Provider，覆盖 OpenAI、DeepSeek、通义千问、月之暗面、火山方舟、Ollama 等兼容接口。后续若某家模型协议不兼容，再单独新增 Provider，不影响划词、OCR、缓存和 UI 层。

模型接口需要支持：

- 流式与非流式两种调用。
- 请求取消，用户关闭窗口或重新生成时中断正在进行的 HTTP 连接。
- 超时控制，避免网络请求长期挂起。
- 错误归一化，将鉴权失败、限流、网络错误、模型不存在等转成统一错误码。
- token 用量记录，便于后续统计成本。

### 6.5 UI 可扩展设计

为后期美化窗口提前做好准备，第一版 UI 不追求复杂视觉，但结构上要保持可扩展。

- 使用设计变量管理颜色、圆角、阴影、字号、间距，避免样式散落在组件里。
- 悬浮窗拆分为 `Header`、`SourceText`、`ResultStream`、`ActionBar`、`StatusBar` 等组件。
- 结果区域使用卡片化数据结构，但第一版保持轻量样式。
- 支持暗色/浅色主题预留，不要求第一版完整开放主题设置。
- 窗口尺寸、最小宽高、最大宽高和滚动区域固定，防止流式输出时布局跳动。
- 将动画控制在窗口出现、加载状态和结果增量更新三类，避免影响性能。

### 6.6 生命周期与资源清理

各环节必须明确生命周期，避免窗口、事件监听、图片缓冲区、请求连接和 OCR 对象泄漏。

```text
Idle
  -> SelectionQuery
  -> CaptureSelecting
  -> OcrRunning
  -> LlmStreaming
  -> Completed
  -> Idle
```

生命周期要求：

- 每次查询生成唯一 `query_id`，后续 OCR、API 请求、缓存写入和 UI 更新都绑定该 ID。
- 新查询开始时，取消上一次未完成的 OCR/API 任务。
- 关闭悬浮窗时，中断流式请求并停止向 UI 推送事件。
- 框选结束后，立即销毁遮罩窗口和鼠标事件监听。
- OCR 完成后，释放截图 bitmap、临时文件、图像流和识别器临时对象。
- 剪贴板读取完成后恢复原剪贴板内容，失败时记录日志但不阻塞解释。
- 历史记录和缓存写入放在后台任务中，失败不影响主流程。
- 应用退出时注销全局快捷键、关闭数据库连接、取消所有 pending request。

建议增加开发期诊断：

- 统计当前活动窗口数量。
- 统计 pending API 请求数量。
- 统计最近一次截图图片大小。
- 在 debug 模式显示 query 生命周期日志。

## 7. 大模型 Prompt 策略

### 7.1 Prompt 实现方式

第一版采用“内置模板 + 自动路由 + 用户可配置”的方式实现 prompt。

```text
输入文本
  -> 文本预处理
  -> 内容类型判断
  -> 选择 prompt 模板
  -> 拼接用户文本和来源信息
  -> 调用模型
  -> 流式展示结果
```

内容类型判断不单独调用模型，优先用本地轻量规则完成：

- 包含股票、财报、估值、指标关键词时，走股票/金融 prompt。
- 英文占比高且较短时，走英语单词/短语 prompt。
- 英文占比高且较长时，走英文句子/段落 prompt。
- 中文短词或未知内容，走通用名词解释 prompt。
- 用户也可以在悬浮窗中手动切换“股票”“英语”“翻译”“通用”模式。

配置策略：

- 内置默认 prompt 存在代码或资源文件中，保证开箱即用。
- 用户自定义 prompt 存入本地设置。
- 每个 prompt 带 `prompt_version`，缓存时一起记录，避免模板更新后误用旧缓存。
- 设置页提供“恢复默认 prompt”按钮。

### 7.2 默认解释 Prompt

```text
你是一个股票和英语学习助手。
用户会提供一段通过划词或截图 OCR 得到的文字。

请先判断内容类型：
- 股票/金融术语
- 英语单词或短语
- 英文句子
- 普通知识概念

请用中文输出，保持简洁清楚：
1. 一句话解释
2. 必要时给出翻译
3. 如果是股票/金融内容，说明用途、例子和常见误区
4. 如果是英语内容，说明语境含义、词性/语法和自然例句
5. 如果 OCR 文字疑似有错，请指出可能的识别问题

不要长篇大论，不确定时直接说明。
```

### 7.3 专业化 Prompt 模板

#### 股票/金融术语 Prompt

```text
你是一个面向股票初学者的金融术语解释助手。
请用中文解释用户提供的股票、财报或金融内容。

输出要求：
1. 一句话解释这个概念
2. 它在股票分析中的用途
3. 给一个简单例子
4. 常见误区或风险
5. 如果它不是金融术语，请直接说明并按通用概念解释

保持简洁，不提供投资建议，不预测涨跌。
```

#### 英语学习 Prompt

```text
你是一个英语学习助手。
请解释用户提供的英文单词、短语或句子。

输出要求：
1. 中文含义
2. 当前语境下的自然理解
3. 词性、语法或搭配说明
4. 一个简单例句
5. 易混淆表达或使用注意

保持简洁，优先帮助用户读懂当前内容。
```

#### 翻译 Prompt

```text
你是一个准确、自然的翻译助手。
请把用户提供的内容翻译成中文。

输出要求：
1. 先给自然中文翻译
2. 必要时解释关键词或句子结构
3. 如果 OCR 结果疑似有错，请指出可能错误

不要扩展无关背景。
```

#### 通用名词解释 Prompt

```text
你是一个通俗概念解释助手。
请用中文解释用户提供的名词、短语或句子。

输出要求：
1. 一句话解释
2. 通俗类比或例子
3. 使用场景
4. 容易误解的点

保持简洁清楚。
```

### 7.4 响应格式建议

```text
一句话：

解释：

例子：

注意：
```

第一版不强制 JSON 输出，优先保证流式体验和自然阅读。

## 8. 性能目标

### 8.1 响应目标

- 快捷键触发到窗口出现：小于 100ms。
- 读取剪贴板文本：小于 200ms。
- 清晰截图 OCR：目标 100-500ms。
- 命中缓存结果：小于 100ms。
- API 首 token：目标 0.5-1.5s，取决于模型和网络。

### 8.2 用户感知优化

- 触发后立即弹窗，避免用户等待空白状态。
- OCR 完成后先显示识别文本。
- 模型请求使用流式输出。
- 常见查询走缓存。
- 默认解释长度控制在 100-200 字。

## 10. 开发里程碑

### 阶段 1：原型验证，预计 3-5 天

- 搭建 Tauri + React 项目。
- 实现设置页和悬浮窗口。
- 实现 OpenAI-compatible API 调用。
- 实现流式输出。
- 实现手动输入文本查询。
- 建立 `LlmProvider` 抽象和 prompt 模板加载机制。
- 建立基础 UI 组件结构和设计变量。

验收标准：

- 能输入一段文字并获得模型解释。
- 能配置 API Key、Base URL 和模型。
- 能流式显示结果。
- 能切换至少两种 prompt 模式，例如通用解释和英语解释。
- 关闭窗口时能中断当前流式请求。

### 阶段 2：划词解释，预计 3-5 天

- 注册全局快捷键。
- 模拟复制并读取剪贴板。
- 恢复剪贴板内容。
- 查询后弹出悬浮解释窗口。
- 加入 SQLite 缓存。
- 补充 query 生命周期管理，保证连续触发时取消上一次未完成请求。

验收标准：

- 在浏览器、PDF、文档中选中文本后可快捷解释。
- 相同文本第二次查询直接命中缓存。
- 连续多次触发查询不会出现多个残留请求或重复窗口。
- 查询结束后剪贴板内容尽量恢复。

### 阶段 3：框选 OCR，预计 5-8 天

- 实现全屏遮罩与区域框选。
- 快捷键触发后，以当前鼠标光标位置作为框选起点。
- 实现选区截图。
- 接入 Windows 系统 OCR。
- OCR 文本清理与错误提示。
- OCR 后自动进入解释流程。
- 完成遮罩窗口、鼠标监听、截图缓冲区、OCR 对象的释放策略。

验收标准：

- 可框选股票软件、网页图片、PDF 扫描文字并识别。
- 识别文本能进入大模型解释流程。
- 按 `Esc` 取消框选时不留下遮罩窗口或事件监听。
- 连续框选 20 次后内存占用无明显持续增长。

### 阶段 4：体验打磨，预计 3-5 天

- 优化悬浮窗位置、大小和快捷操作。
- 增加复制结果、重新生成、收藏。
- 增加历史记录。
- 完善错误提示。
- 打包安装包。
- 加入 debug 模式生命周期日志和基础性能统计。

验收标准：

- 工具可日常使用。
- 设置、快捷键、缓存、OCR、API 调用流程稳定。
- 资源清理、请求取消、窗口关闭行为经过压力测试。

## 11. 风险与应对

### 11.1 OCR 准确率风险

风险：截图内容模糊、字体太小、背景复杂时识别错误。

应对：

- 第一版提示用户确认 OCR 文本。
- 提供“编辑识别文本后再解释”功能。
- 后续增加 PaddleOCR 插件作为高精度选项。

### 11.2 API 延迟风险

风险：大模型响应速度取决于网络和服务商。

应对：

- 使用流式输出。
- 默认短回答。
- 增加缓存。
- 支持用户选择更快的小模型。

### 11.3 系统权限风险

风险：截图、全局快捷键、剪贴板操作可能受系统权限影响。

应对：

- Windows 先行，减少跨平台复杂度。
- 设置页提供权限检测。
- 遇到权限问题给出明确提示。

### 11.4 剪贴板打扰风险

风险：模拟复制可能覆盖用户剪贴板。

应对：

- 读取前保存原剪贴板内容。
- 查询后恢复。
- 若恢复失败，显示轻提示。

### 11.5 内存泄漏与资源残留风险

风险：截图框选、OCR、流式请求和悬浮窗口都属于短生命周期任务，如果事件监听、图片缓冲区、HTTP 连接或窗口引用没有及时释放，长时间使用后可能造成内存持续增长。

应对：

- 每次查询使用独立 `query_id` 管理状态。
- 用统一的取消令牌管理 OCR 和 API 请求。
- 窗口关闭、重新生成、新查询开始、应用退出时都触发清理。
- 截图数据优先使用内存对象，避免落盘；如需临时文件，必须在任务结束后删除。
- debug 模式记录资源创建和释放日志。
- 阶段 3 起加入连续框选和连续查询压力测试。

## 12. 工程质量要求

### 12.1 状态管理

- 全局只保留一个活动查询状态。
- 每个查询包含 `query_id`、source、mode、stage、abort_handle、created_at。
- UI 只接受当前 `query_id` 的事件，过期事件直接丢弃。
- 避免流式返回晚到后覆盖新查询结果。

### 12.2 资源清理 Checklist

- 全局快捷键：应用退出时注销。
- 鼠标监听：框选完成或取消后移除。
- 遮罩窗口：框选完成或取消后销毁。
- 截图 bitmap：OCR 完成后释放。
- OCR 对象：识别完成后释放引用。
- HTTP 流式连接：窗口关闭、重新生成、新查询时取消。
- 数据库连接：应用退出时关闭。
- 定时器和防抖任务：组件卸载时清理。

### 12.3 压力测试建议

- 连续划词查询 50 次，观察请求取消和窗口状态。
- 连续框选 OCR 20 次，观察内存曲线。
- 在 API 慢响应时关闭窗口，确认请求被中断。
- 在流式输出中立即重新生成，确认旧响应不会继续写入 UI。
- OCR 失败、API 失败、缓存写入失败分别测试错误恢复。

## 16. 总结

本项目应采用“轻客户端 + 大模型 API”的路线。本地端专注于快速获取文本、OCR、缓存和展示，大模型负责解释、翻译和语境理解。

第一版优先面向 Windows，实现划词解释和框选 OCR 两条主路径。只要交互足够快、解释足够准、窗口足够轻，就能满足股票和英语学习中的高频需求。后续再逐步扩展生词本、术语库、浏览器插件和跨平台能力。
